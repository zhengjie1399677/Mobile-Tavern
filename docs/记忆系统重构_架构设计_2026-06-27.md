## 一、背景与目标

### 1.1 项目背景

Mobile-Tavern 定位为 SillyTavern 生态兼容的纯移动端角色扮演容器。当前已有两套独立的记忆机制：

- **AutoSummaryService**：滚动窗口 LLM 摘要 + 5 条正则抽离（location/time/condition/inventory/bonding）
- **TableMemoryService**：AI 指令式 CRUD 结构化表格

### 1.2 核心痛点

经过对生产环境模型调用数据（DeepSeek 38.7%、Gemini 19%、Claude 12%、GLM 6.6%、其他 24%）的分析与多轮讨论，识别出以下核心痛点：

| 痛点 | 表现 | 根因 |
|---|---|---|
| 职责重叠 | AutoSummary 的状态抽离与 TableMemory 功能冲突 | 两套系统都在做"状态提取"，触发机制不同但产出重叠 |
| 深度记忆缺失 | 角色记不住早期具体细节 | 当前只有"压缩/抽象"，无"细节召回" |
| 摘要质量不稳定 | 5 条正则强依赖 LLM 输出格式 | 正则方案天然脆弱 |
| 大对象污染 | `sessions.messages` 数组全量反序列化 | 违反 AGENTS.md 准则一"拒绝大对象污染" |
| 模型兼容脆弱 | 不支持 JSON Schema 的模型直接报错 | 缺少能力识别与降级机制 |

### 1.3 设计目标

1. **统一记忆服务**：合并 AutoSummaryService 与 TableMemoryService 为单一 `MemoryService`
2. **检索式深度记忆**：原始对话永久保留 + 标签召回相关片段
3. **零额外配置**：小白用户无需配置即可使用，复用主对话模型
4. **模型兼容降级**：自动识别模型能力，不支持时平滑降级
5. **SillyTavern 生态兼容**：角色卡可零侵入运行，记忆能力为可选增强
6. **物理分轨隔离**：原始对话、索引、状态表分别独立存储，不污染主表

## 二、设计原则

本设计严格遵循 AGENTS.md 中的核心行为准则：

| 准则 | 在本设计中的体现 |
|---|---|
| 准则一：物理分轨隔离 | messages / memory_dict / sessions 三 Store 分离，sessions 表瘦身 |
| 准则二：SillyTavern 兼容 | 角色卡词典可选，无词典角色卡零侵入运行；所有记忆机制可关闭 |
| 准则三：外部接口防腐 | 模型能力表 + 参数清洗层 cleanLLMParams |
| 准则五：向前兼容 | 老角色卡无 memory 字段照常工作；存量数据平滑迁移 |
| 准则六：单文件 ≤1000 行 | MemoryService 拆为 6 个子模块 + barrel |
| 准则八/十：物理隔离开发 | 新建 `src/kernel/services/memory/` 沙盒，TDD 单兵验证 |
| 准则二豁免：模型能力元数据 | 模型能力表硬编码在底层，属必要技术元数据（类似驱动兼容表） |

## 三、整体架构

### 3.1 分层认知记忆架构

```
┌──────────────────────────────────────────────────────────┐
│                  MemoryService（统一记忆服务）            │
├──────────────────────────────────────────────────────────┤
│                                                          │
│ 【存储层】IndexedDB 物理分轨（不换库）                   │
│  ├─ messages Store（新增）：原始对话永久保留             │
│  │   index: sessionId, createdAt, tags（多值索引）       │
│  ├─ sessions Store（瘦身）：元信息+摘要+状态表           │
│  ├─ memory_dict Store（新增）：会话级自动学习词典        │
│  │   { sessionId, entity, type, firstSeenMsgId, count }  │
│  └─ 现有 characters/lorebooks/worldbooks 不动            │
│                                                          │
│ 【抽取层】异步三级降级（绝不阻塞主对话流）              │
│  ├─ L0 主对话顺便抽取（默认）                            │
│  │   system prompt 加入抽取指令，LLM 在回复末尾输出      │
│  │   <memory>JSON</memory>，端侧解析                     │
│  ├─ L1 词典匹配（兜底）                                  │
│  │   会话级自动词典 + 角色卡可选词典，AC 自动机匹配      │
│  └─ L2 仅存原文（最终兜底）                              │
│     L0/L1 均未命中时，仅写入 messages Store 不抽标签     │
│                                                          │
│ 【召回层】Prompt 装配时（memoryRecallMiddleware）        │
│  ├─ 最近 N 轮原文（recentTurns，默认 5）                 │
│  ├─ 历史摘要时间线（SummaryCard[]，宏观脉络）            │
│  └─ 标签召回 top-K 相关片段（默认 top-3，微观细节）      │
│                                                          │
│ 【状态层】合并自 TableMemory（MemoryStateTable 子模块）  │
│  └─ 预定义 4 表（物品/关系/位置/任务）+ 角色卡可扩展     │
│                                                          │
│ 【摘要层】瘦身自 AutoSummary（MemorySummary 子模块）     │
│  └─ 只做剧情时间线摘要，砍掉 5 条正则状态抽离           │
│                                                          │
│ 【兼容层】模型能力表 + 参数清洗（ModelCapabilityRegistry）│
│  ├─ 已知模型硬编码能力元数据（豁免准则二）              │
│  ├─ 未知模型保守降级                                    │
│  └─ 运行时错误自愈（缓存学习到的能力）                  │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 3.2 记忆装配到 Prompt 的结构

```
LLM 装配 Prompt 时的记忆区块（由 PromptService 读取）:

【相关记忆】（标签召回，微观细节，top-3）
  - 第 42 轮: 酒馆老板老张递来梅子酒...
  - 第 87 轮: 老张提到他女儿...

【前情提要】（AutoSummary 摘要，宏观脉络）
  - 第 1 幕: 主角初到酒馆，结识老张
  - 第 2 幕: 老张委托主角寻找失散的女儿

【状态快照】（TableMemory，当前世界状态）
  - 物品: 梅子酒 x1, 祖传玉佩 x1
  - 关系: 老张 (好感度 75/100)

【近期对话】（recentTurns，原文，默认 5 轮）
  - 最近 5 轮原文
```

三层互补：召回给"细节"，摘要给"脉络"，状态给"现在"，近期给"上下文"。

## 四、数据模型设计

### 4.1 IndexedDB Store 结构

#### 4.1.1 messages Store（新增）

存储所有原始对话消息，按 sessionId 隔离，永久保留。

```typescript
interface MessageRecord {
  id: string;              // 消息唯一 ID（uuid）
  sessionId: string;       // 所属会话 ID
  role: 'user' | 'assistant' | 'system';
  content: string;         // 原始消息内容（剥离 <memory> 标签后）
  createdAt: number;       // 创建时间戳
  turnIndex: number;       // 在会话中的轮次序号
  tags?: string[];         // 抽取的实体标签（如 ["老张", "梅子酒", "酒馆"]）
  extractSource?: 'llm' | 'dict' | 'none';  // 标签来源
  metadata?: {
    modelUsed?: string;    // 生成该消息的模型
    tokenCount?: number;
  };
}

// 索引设计
indexes: {
  sessionId: 'sessionId',          // 按会话查询
  createdAt: 'createdAt',          // 按时间排序
  'sessionId,createdAt': ['sessionId', 'createdAt'],  // 复合索引，分页查询
  tags: 'tags',                    // 多值索引，按标签召回
}
```

#### 4.1.2 memory_dict Store（新增）

会话级自动学习词典，记录从对话中涌现的实体。

```typescript
interface MemoryDictEntry {
  id: string;              // `${sessionId}:${entity}` 复合键
  sessionId: string;
  entity: string;          // 实体名（如 "老张"）
  aliases: string[];       // 别名（如 ["张老板", "酒馆老板"]）
  type: 'character' | 'location' | 'item' | 'organization' | 'concept';
  firstSeenMsgId: string;  // 首次出现的消息 ID
  firstSeenTurn: number;
  count: number;           // 出现次数（用于热度排序）
  createdAt: number;
  updatedAt: number;
}

indexes: {
  sessionId: 'sessionId',
  'sessionId,type': ['sessionId', 'type'],
  entity: 'entity',
}
```

#### 4.1.3 sessions Store（瘦身重构）

从 session 记录中剥离 messages 数组，只保留元信息与摘要/状态。

```typescript
interface ChatSession {
  id: string;
  characterId: string;
  title: string;
  createdAt: number;
  updatedAt: number;
  lastMessageId?: string;          // 指向 messages Store 中最后一条
  lastMessagePreview?: string;     // 最后一条消息预览（用于列表展示，≤100 字）
  messageCount: number;            // 消息总数
  
  // 记忆相关（保留在 session 中，因为体积小且高频读取）
  summaries: SummaryCard[];        // 剧情时间线摘要
  lastSummarizedMessageId?: string;
  tableMemory: TableMemorySheet[]; // 状态表（合并自 TableMemory）
  
  // 兼容字段（迁移期保留，迁移完成后可移除）
  messages?: Message[];            // 仅在迁移期间存在，迁移后置空
}

// 索引保持不变
indexes: {
  characterId: 'characterId',
  createdAt: 'createdAt',
}
```

#### 4.1.4 现有 Store 不变

- `characters`：角色卡，无变更
- `settings`：全局设置，新增记忆相关配置项（见第十一章）
- `lorebooks` / `worldbooks`：无变更

### 4.2 数据库版本升级

IndexedDB schema 版本从当前版本 +1 升级，升级脚本在 `MemoryStorage.init()` 中实现：

```typescript
// 伪代码
function upgradeDB(event: IDBVersionChangeEvent) {
  const db = event.target.result;
  
  if (event.oldVersion < TARGET_VERSION) {
    // 1. 创建 messages Store
    if (!db.objectStoreNames.contains('messages')) {
      const messagesStore = db.createObjectStore('messages', { keyPath: 'id' });
      messagesStore.createIndex('sessionId', 'sessionId', { unique: false });
      messagesStore.createIndex('createdAt', 'createdAt', { unique: false });
      messagesStore.createIndex('tags', 'tags', { unique: false, multiEntry: true });
      messagesStore.createIndex('sessionId_createdAt', ['sessionId', 'createdAt'], { unique: false });
    }
    
    // 2. 创建 memory_dict Store
    if (!db.objectStoreNames.contains('memory_dict')) {
      const dictStore = db.createObjectStore('memory_dict', { keyPath: 'id' });
      dictStore.createIndex('sessionId', 'sessionId', { unique: false });
      dictStore.createIndex('entity', 'entity', { unique: false });
    }
    
    // 3. sessions Store 索引保持不变，字段在迁移阶段处理
  }
}
```

## 五、服务拆分设计

### 5.1 目录结构

按 AGENTS.md 准则六"单文件 ≤1000 行"，MemoryService 拆为子模块：

```
src/kernel/services/memory/
├── MemoryService.ts          # 主服务入口，~200 行
│                              # 实现 IKernelService，协调子模块
├── MemoryExtractor.ts        # L0 LLM 抽取 + L1 词典匹配，~450 行
├── MemoryRecall.ts           # 召回逻辑（倒排索引查询 + top-K），~250 行
├── MemoryStorage.ts          # IndexedDB 读写封装（messages/dict Store），~350 行
├── MemoryStateTable.ts       # 合并自 TableMemoryService，~350 行
├── MemorySummary.ts          # 合并自 AutoSummaryService（瘦身版），~300 行
├── ModelCapabilityRegistry.ts # 模型能力表 + 参数清洗，~250 行
├── MemoryStreamParser.ts     # 流式 <memory> 标签解析器，~150 行
├── types.ts                  # 内部类型定义，~120 行
└── index.ts                  # barrel 文件，重新导出公共 API
```

### 5.2 主服务接口契约

```typescript
// src/kernel/services/memory/MemoryService.ts
import { KernelServices, IKernelService, IKernel } from '../../types';
import { MemoryExtractor } from './MemoryExtractor';
import { MemoryRecall } from './MemoryRecall';
import { MemoryStorage } from './MemoryStorage';
import { MemoryStateTable } from './MemoryStateTable';
import { MemorySummary } from './MemorySummary';

export class MemoryService implements IKernelService {
  readonly name = KernelServices.Memory;
  readonly dependencies = [KernelServices.Database, KernelServices.LLM] as const;

  private extractor!: MemoryExtractor;
  private recall!: MemoryRecall;
  private storage!: MemoryStorage;
  private stateTable!: MemoryStateTable;
  private summary!: MemorySummary;

  async init(kernel: IKernel, signal?: AbortSignal): Promise<void> {
    const db = kernel.getService(KernelServices.Database);
    const llm = kernel.getService(KernelServices.LLM);

    // 1. 初始化存储层（确保 messages/dict Store 已建）
    this.storage = new MemoryStorage(db);
    await this.storage.init(signal);

    // 2. 初始化子模块
    this.extractor = new MemoryExtractor(this.storage, llm);
    this.recall = new MemoryRecall(this.storage);
    this.stateTable = new MemoryStateTable(this.storage);
    this.summary = new MemorySummary(this.storage, llm);

    // 3. 绑定 AbortSignal 到所有子模块的定时器/轮询
    signal?.addEventListener('abort', () => this.handleAbort());
  }

  async destroy(signal?: AbortSignal): Promise<void> {
    // 传递 signal 给所有子模块，确保异步任务全部中止
    await Promise.allSettled([
      this.extractor?.destroy(signal),
      this.summary?.destroy(signal),
    ]);
  }

  // 公共 API（供中间件调用）
  getExtractors() { return this.extractor; }
  getRecall() { return this.recall; }
  getStateTable() { return this.stateTable; }
  getSummary() { return this.summary; }
  getStorage() { return this.storage; }

  private handleAbort() {
    // 中止所有进行中的异步任务
  }
}
```

### 5.3 子模块职责

| 子模块 | 职责 | 关键方法 |
|---|---|---|
| MemoryStorage | IndexedDB 读写封装 | `appendMessage()`, `getMessageById()`, `getMessagesBySession()`, `getMessagesByTag()`, `upsertDictEntry()`, `getDictBySession()` |
| MemoryExtractor | L0 LLM 抽取 + L1 词典匹配 | `extractFromLLMResponse()`, `extractByDict()`, `scheduleExtraction()` |
| MemoryRecall | 召回相关历史片段 | `recall(message, topK)`, `recallByTags(tags, topK)` |
| MemoryStateTable | 状态表 CRUD（合并自 TableMemory） | `parseAICommand()`, `updateRow()`, `getSheet()`, `initDefaultSheets()` |
| MemorySummary | 剧情摘要（瘦身自 AutoSummary） | `checkAndSummarize()`, `generateSummary()` |
| ModelCapabilityRegistry | 模型能力识别 + 参数清洗 | `getCapabilities(model)`, `cleanLLMParams(model, params)`, `updateCapabilities(model, caps)` |
| MemoryStreamParser | 流式 `<memory>` 标签解析 | `onChunk(chunk)`, `finalize()` |

## 六、中间件设计

### 6.1 中间件清单

合并原有 `autoSummaryMiddleware`（priority 70）与 `tableMemoryMiddleware`（priority 100），新增召回中间件：

| 中间件 | 方向 | Priority | 职责 |
|---|---|---|---|
| `memoryRecallMiddleware` | input | 30 | 组装 Prompt 前召回相关历史，注入 `ctx.relevantMemories` |
| `memoryExtractMiddleware` | output | 80 | AI 回复完成后异步触发抽取（合并原 summary + tableMemory + 标签抽取） |

### 6.2 中间件优先级协调

```
input 阶段:
  priority 10: lorebook 关键词触发
  priority 20: worldbook 注入
  priority 30: memoryRecallMiddleware（新增）  ← 在 PromptService 之前
  priority 50: PromptService 组装

output 阶段:
  priority 80: memoryExtractMiddleware（新增，合并 summary + tableMemory + 抽取）
```

### 6.3 中间件实现伪代码

```typescript
// memoryRecallMiddleware（input，priority 30）
const memoryRecallMiddleware: IMiddleware = {
  priority: 30,
  phase: 'input',
  handler: async (ctx, next) => {
    const memoryService = kernel.getService(KernelServices.Memory);
    const topK = getUserSettings().memoryRecallTopK ?? 3;
    
    // 1. 从当前消息提取查询标签
    const queryTags = memoryService.getExtractors().extractByDict(
      ctx.sessionId, ctx.message
    );
    
    // 2. 召回相关历史片段
    const recalled = await memoryService.getRecall().recallByTags(
      ctx.sessionId, queryTags, topK
    );
    
    // 3. 注入到上下文，PromptService 会读取并注入 Prompt
    ctx.relevantMemories = recalled;
    
    await next();
  },
};

// memoryExtractMiddleware（output，priority 80）
const memoryExtractMiddleware: IMiddleware = {
  priority: 80,
  phase: 'output',
  handler: async (ctx, next) => {
    await next();  // 先让响应返回给用户
    
    // 异步触发抽取，不阻塞响应
    const memoryService = kernel.getService(KernelServices.Memory);
    memoryService.getExtractors().scheduleExtraction({
      sessionId: ctx.sessionId,
      message: ctx.response,
      turnIndex: ctx.turnIndex,
    }).catch(err => {
      kernel.log.warn('Memory extraction failed', err);
    });
    
    // 异步触发摘要检查（原 AutoSummary 逻辑）
    memoryService.getSummary().checkAndSummarize(ctx.sessionId).catch(err => {
      kernel.log.warn('Auto summary failed', err);
    });
    
    // 异步处理 AI 指令式状态表更新（原 TableMemory 逻辑）
    memoryService.getStateTable().parseAICommand(
      ctx.sessionId, ctx.response
    ).catch(err => {
      kernel.log.warn('State table update failed', err);
    });
  },
};
```

## 七、记忆抽取机制

### 7.1 抽取时机

**默认策略**：每轮异步抽取（不阻塞主对话流）。

```
用户发消息
     ↓
LLM 流式回复（主流程，1 次 API 调用，复用主对话模型）
     ↓
回复完成 → 立即返回给用户（用户无感延迟）
     ↓
【异步触发抽取任务】
     ↓
检查抽取队列状态:
  ├─ 上一轮抽取未完成 → 跳过本轮（节流，防堆积）
  ├─ 队列空闲 → 入队执行
  └─ 队列满（>3 个待执行）→ 丢弃最旧的，入队新的
     ↓
执行抽取（从主对话回复中解析 <memory> 标签）
     ↓
写入 messages Store + memory_dict Store
```

### 7.2 L0 主对话顺便抽取（默认机制）

#### 7.2.1 System Prompt 注入

PromptService 在组装 system prompt 时，追加抽取指令（可由角色卡覆盖）：

```
[原有角色设定]
[原有世界观设定]
...

<memory_extract_instruction>
在回复的最末尾，用 <memory></memory> 标签输出从本轮对话抽取的记忆实体。
标签内必须是一个合法 JSON，格式如下:
{
  "entities": [
    {"name": "实体名", "type": "character|location|item|organization|concept", "first_seen": true|false}
  ],
  "events": [
    {"summary": "事件简述（≤50字）", "participants": ["参与实体名"]}
  ]
}

规则:
1. 只抽取"具体名词"，不抽抽象描述（"老张"✓，"友善的老板"✗）
2. 实体名用对话原文的称呼
3. events.summary 不超过 50 字
4. first_seen=true 表示这是首次出现的实体
5. 如果本轮没有新实体或事件，输出空对象: <memory>{"entities":[],"events":[]}</memory>
</memory_extract_instruction>
```

#### 7.2.2 模型能力自适应

根据 ModelCapabilityRegistry 识别模型能力：

| 模型能力 | 抽取方式 |
|---|---|
| 支持 JSON Schema | system prompt 注入抽取指令 + `response_format: json_schema` 强制结构化（但 LLM 仍输出在 `<memory>` 标签内，端侧解析） |
| 不支持 JSON Schema | 仅 system prompt 注入抽取指令，靠 LLM 自觉 + 端侧容错解析 |

#### 7.2.3 流式解析器

LLM 流式输出时，`<memory>` 标签可能跨 chunk 到达。使用状态机解析：

```typescript
// src/kernel/services/memory/MemoryStreamParser.ts
class MemoryStreamParser {
  private buffer = '';
  private inMemoryTag = false;
  private memoryContent = '';
  
  /**
   * 处理一个流式 chunk
   * @returns displayText: 应展示给用户的文本（已剥离 <memory> 标签）
   *          memoryContent: 完整的 <memory> 内容（仅在标签闭合时返回）
   */
  onChunk(chunk: string): { displayText: string; memoryContent?: string } {
    this.buffer += chunk;
    
    if (!this.inMemoryTag) {
      const openIdx = this.buffer.indexOf('<memory>');
      if (openIdx === -1) {
        // 还没遇到 <memory>，全部展示（但保留末尾可能的不完整标签）
        const safeEnd = this.findSafeDisplayEnd();
        const result = this.buffer.slice(0, safeEnd);
        this.buffer = this.buffer.slice(safeEnd);
        return { displayText: result };
      }
      // 遇到 <memory>，展示之前的内容
      const before = this.buffer.slice(0, openIdx);
      this.buffer = this.buffer.slice(openIdx + 8); // 跳过 <memory>
      this.inMemoryTag = true;
      return { displayText: before };
    }
    
    // 在 <memory> 内部，寻找 </memory>
    const closeIdx = this.buffer.indexOf('</memory>');
    if (closeIdx === -1) {
      // 还没结束，继续缓冲（不展示给用户）
      this.memoryContent += this.buffer;
      this.buffer = '';
      return { displayText: '' };
    }
    // 提取完整 memory 内容
    this.memoryContent += this.buffer.slice(0, closeIdx);
    this.buffer = this.buffer.slice(closeIdx + 9); // 跳过 </memory>
    this.inMemoryTag = false;
    const result = { displayText: '', memoryContent: this.memoryContent };
    this.memoryContent = '';
    return result;
  }
  
  /**
   * 找到安全的展示截止位置（避免截断可能的开标签）
   */
  private findSafeDisplayEnd(): number {
    // 检查 buffer 末尾是否可能是 <memory 的前缀
    for (let i = Math.max(0, this.buffer.length - 8); i < this.buffer.length; i++) {
      if (this.buffer[i] === '<') {
        const tail = this.buffer.slice(i);
        if ('<memory>'.startsWith(tail)) {
          return i; // 截断到这里，剩余的留到下次
        }
      }
    }
    return this.buffer.length;
  }
  
  finalize(): { displayText: string; memoryContent?: string } {
    // 流结束时，如果还在 <memory> 内部，尝试解析已缓冲内容
    if (this.inMemoryTag && this.memoryContent) {
      return { displayText: '', memoryContent: this.memoryContent };
    }
    return { displayText: this.buffer };
  }
}
```

#### 7.3.2 AC 自动机匹配

使用 Aho-Corasick 算法一次扫描匹配所有词典实体，性能 O(M+Z)：

```typescript
// 伪代码
import AhoCorasick from 'aho-corasick';

function extractByDict(
  message: string,
  dict: MemoryDictEntry[]
): string[] {
  const allTerms = dict.flatMap(entry => [entry.entity, ...entry.aliases]);
  const ac = new AhoCorasick(allTerms);
  const matches = ac.search(message);
  
  // 去重 + 返回命中的实体名
  const hitEntities = new Set<string>();
  for (const [endPos, terms] of matches) {
    for (const term of terms) {
      hitEntities.add(term);
    }
  }
  return Array.from(hitEntities);
}
```

**性能测算**：
- 词典 500 条 × 消息 1000 字符 → < 2ms
- 词典 2000 条 × 消息 1000 字符 → < 5ms
- 完全无感

### 7.4 L2 仅存原文（最终兜底）

当 L0 和 L1 均未命中时，仅将消息原文写入 messages Store，`tags` 字段为空数组，`extractSource` 为 `'none'`。

后续召回时该消息仍可被其他消息的标签间接召回（如果其他消息提到了相同实体）。

```typescript
    private storage: MemoryStorage,
    private llm: ILLMService
  ) {}

   * 检查并触发摘要
   */
  async checkAndSummarize(sessionId: string): Promise<void> {

    const session = await this.storage.getSession(sessionId);
    
    // 1. 获取未总结的消息（从 messages Store 查询）

      max_tokens: 500,

        type: 'json_schema',
        json_schema: {
          name: 'summary_card',
```

```typescript
class ModelCapabilityRegistry {
  private static runtimeCache: Map<string, ModelCapabilities> = new Map();
  
  /**
   * 获取模型能力
   * 优先级: 运行时缓存 > 已知表 > 默认保守值
   */
  static getCapabilities(modelId: string): ModelCapabilities {
    // 1. 查运行时缓存（错误自愈学到的）
    if (this.runtimeCache.has(modelId)) {
      return this.runtimeCache.get(modelId)!;
    }
    
    // 2. 查已知表（按前缀匹配）
    for (const [prefix, caps] of Object.entries(KNOWN_MODEL_CAPABILITIES)) {
      if (modelId.toLowerCase().startsWith(prefix)) {
        return caps;
      }
    }
    
    // 3. 默认保守值
    return DEFAULT_CAPABILITIES;
  }
  
  /**
   * 清洗 LLM 参数（防腐层）
   * 移除模型不支持的参数，避免 400 错误
   */
  static cleanLLMParams(modelId: string, params: LLMParams): LLMParams {
    const caps = this.getCapabilities(modelId);
    return {
      ...params,
      top_k: caps.supportsTopK ? params.top_k : undefined,
      top_p: caps.supportsTopP ? params.top_p : undefined,
      temperature: caps.supportsTemperature ? params.temperature : undefined,
      response_format: caps.supportsJsonSchema ? params.response_format : undefined,
      functions: caps.supportsFunctionCalling ? params.functions : undefined,
    };
  }
  
  /**
   * 运行时更新能力（错误自愈后调用）
   */
  static updateCapabilities(modelId: string, patch: Partial<ModelCapabilities>): void {
    const current = this.getCapabilities(modelId);
    this.runtimeCache.set(modelId, { ...current, ...patch });
    // 持久化到 localStorage，重启后保留
    this.persistToLocalStorage();
  }
  
  /**
   * 识别"参数不支持"错误
   */
  static isUnsupportedParamError(error: any): { supported: false; param: string } | null {
    const errorMsg = String(error?.message || error || '');
    const paramPatterns = [
      { param: 'top_k', pattern: /top_k|topK/i },
      { param: 'top_p', pattern: /top_p|topP/i },
      { param: 'response_format', pattern: /response_format|json_schema/i },
      { param: 'function_call', pattern: /function_call|tools/i },
    ];
    
    for (const { param, pattern } of paramPatterns) {
      if (pattern.test(errorMsg)) {
        return { supported: false, param };
      }
    }
    return null;
  }
}
```

```typescript
      // 缓存能力，下次不再发该参数
      ModelCapabilityRegistry.updateCapabilities(model, {
        [`supports${capitalize(unsupported.param)}`]: false,
      } as any);
      
      // 移除该参数重试（最多 1 次）
      const retryParams = { ...cleanedParams, [unsupported.param]: undefined };
      return await rawCallLLM(model, retryParams);
    }
    throw error;
  }
}
```

## 十四、存量数据迁移方案

### 14.1 迁移策略

采用**渐进式迁移**，确保老用户平滑升级：

#### 阶段 1：版本升级时自动迁移（一次性）

```typescript
// 在 MemoryStorage.init() 中执行
async function migrateSessionsToMessagesStore(): Promise<void> {
  const db = getDB();
  const sessions = await db.getAll('sessions');
  
  for (const session of sessions) {
    if (session.messages && session.messages.length > 0) {
      // 1. 将 messages 数组写入 messages Store
      for (let i = 0; i < session.messages.length; i++) {
        const msg = session.messages[i];
        await db.add('messages', {
          id: msg.id || generateUUID(),
          sessionId: session.id,
          role: msg.role,
          content: msg.content,
          createdAt: msg.createdAt || Date.now(),
          turnIndex: i,
          tags: [],  // 存量消息不抽取标签，仅原文保留
          extractSource: 'none',
        });
      }
      
      // 2. 更新 session 元信息
      session.lastMessageId = session.messages.at(-1)?.id;
      session.lastMessagePreview = session.messages.at(-1)?.content?.slice(0, 100);
      session.messageCount = session.messages.length;
      session.messages = undefined;  // 清空大数组
      
      await db.put('sessions', session);
    }
  }
}
```

#### 阶段 2：后台异步补抽标签（可选）

存量消息初始无标签。可在 App 空闲时异步补抽：

```typescript
async function backfillTagsInBackground(): Promise<void> {
  const messagesWithoutTags = await getMessagesByExtractSource('none');
  for (const msg of messagesWithoutTags) {
    // 用 L1 词典匹配补标签（不调 LLM，零成本）
```

### 14.2 迁移风险与对策

| 风险 | 对策 |
|---|---|
| 迁移过程中 App 崩溃 | 迁移在 IndexedDB 事务中执行，崩溃自动回滚 |
| 大量数据迁移耗时 | 分批迁移（每次 100 条），避免阻塞主线程 |
| 老版本回滚兼容 | session.messages 字段保留（迁移后置空），老版本读取到空数组不会崩溃 |
| 迁移失败 | 记录失败会话 ID，下次启动重试 |

## 十五、生命周期资源回收

按 AGENTS.md 准则十，所有子模块在 `destroy(signal)` 时必须清理资源：

| 子模块 | 需回收的资源 | 回收方式 |
|---|---|---|
| MemoryExtractor | 抽取队列定时器、进行中的异步任务 | `signal.addEventListener('abort', ...)` 中止 |
| MemorySummary | 摘要 LLM fetch、滚动触发定时器 | AbortController 中止 fetch，clearTimeout |
| MemoryStorage | 进行中的 IndexedDB 事务 | 事务自动提交/回滚，无需手动处理 |
| MemoryRecall | 无异步资源 | 无需处理 |
| MemoryStateTable | 无异步资源 | 无需处理 |
| ModelCapabilityRegistry | localStorage 写入定时器 | clearTimeout |

```typescript
// MemoryService.destroy 实现
async destroy(signal?: AbortSignal): Promise<void> {
  const destroySignal = signal ?? new AbortController().signal;
  
  await Promise.allSettled([
    this.extractor?.destroy(destroySignal),
    this.summary?.destroy(destroySignal),
  ]);
  
  // storage/recall/stateTable 无需异步销毁
  this.runtimeCache.clear();
}
```

## 十七、风险评估与对策

| 风险 | 等级 | 对策 |
|---|---|---|
| LLM 不输出 `<memory>` 标签 | 中 | L1 词典匹配兜底，最终 L2 仅存原文 |
| LLM 输出格式错误 | 中 | zod 校验 + 三级降级 |

| 存量数据迁移失败 | 中 | 事务回滚 + 失败重试 + 老版本回滚兼容 |
| 模型能力表过时 | 低 | 运行时错误自愈 + 用户可手动覆盖 |
| 召回 token 膨胀 | 中 | top-K 限制（默认 3）+ 单条截断（≤500 字） |
| 异步抽取堆积 | 中 | 节流队列（最多 3 个待执行）+ 丢弃最旧 |
| 角色卡词典冲突 | 低 | 角色卡词典仅作为冷启动，会话级自动词典优先 |
