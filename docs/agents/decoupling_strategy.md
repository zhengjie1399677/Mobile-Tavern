# 超大规模扩展性与极致底座解耦战略

> [!IMPORTANT]
> **此文件为 Mobile Tavern 行为指导手册的子规范，定义了底座解耦、状态管理以及向前兼容的具体细则。**

---

### 1. 长效架构与微服务演进警示
* **大单体崩溃防御**：现有的轻量级紧凑单体结构（如万行级的 UI 业务绑定、大对象全量写入 IndexedDB 机制）在面对超大规模特性时必将崩溃。在构思新功能时，必须首先权衡其对底层架构扩展性的侵蚀，杜绝“为加功能区域加代码”的短视行为。
* **极致微服务与解耦**：核心业务必须从上帝 Hook（如 `useChat`）中彻底解耦和退化，具体的网络通信、视频控制、音频渲染、实时 WebRTC 交互、插件沙盒等，必须全部下沉到独立的切面微服务（Service）或动态加载的微模块中。

### 2. 物理层数据严格解耦与隔离
* **拒绝大对象污染**：严禁为了图一时省事而将高频、大容量的数据（例如：图片 Base64 字符串、大段背景设定文本、非核心大字段）直接塞入高频读取的全局基础配置表（如 `UserSettings` / `localDB` 的主 settings 记录）中。
* **分轨存储**：必须将其分流至独立的数据库存储单元（如单独的数据库键、专用的 IndexedDB Store 或独立的表），在需要时通过 ID 异步检索，保障主应用初始化和核心读取流程绝对轻量，防止反序列化延时引发的锁死与白屏。
* **绝对物理分轨隔离**：严禁将任何高频交互、非关系型的数据（如音视频状态、插件上下文、实时同步信令）污染核心数据库 settings 或 sessions 表。必须严格使用分轨存储与增量键值存取，确保 App 底层框架无论扩展到何种复杂度，初始化与核心渲染流水线始终在毫秒级内完成。

### 3. 外部接口防腐隔离（Anti-Corruption Layer）
* **输入清洗**：凡是接收自外部接口（包括大模型 API 响应、中转代理响应、第三方导入的角色卡/设定集数据等）的输入，必须在系统入口处统一通过高内聚的纯函数（如 `cleanRequestPayload`）进行数据清洗和结构转换。
* **隔离泄露**：严禁允许未经清洗的非标参数、第三方脏数据或临时兼容逻辑直接渗透、污染到核心逻辑层（如 `useChat`）与数据库物理存储层，必须建立坚实的接口防护隔离带。

### 4. 状态管理边界限制与防副作用泄漏
* **单一职责状态流动**：各个功能模块 of 内部状态（如配置编辑、临时选框等）应尽可能限制在局部 Context、局部 Hook 或组件内部处理，严禁滥用全局 Context（如 AppContext）扩散非核心状态。
* **副作用隔离**：在编写 `useEffect`、事件响应器或订阅逻辑时，必须精确指明依赖项，避免多余的状态变动在整个系统树上引发多重“雪崩式”级联渲染与未捕获的运行时空指针异常。

### 5. 向前兼容与接口降级设计（Fallback）
* **接口变更防御**：在重构、扩展或修改已有的 Props、数据 Model、配置字段或 API 契约时，严禁粗暴地对正在使用的属性进行删除或不向后兼容的类型重定义。
* **默认容错兜底**：凡是涉及参数传递和数据库加载的数据，必须提供可靠的 Fallback（兜底降级方案）与初始化默认值，确保老版本用户在使用新系统时能够无缝升级并平滑容错，不会因为缺少某个新字段而在渲染时崩溃。

### 6. 面向模块化/服务化的轻量化开发
* **拒绝一时贪快的硬编码**：在编写或修改业务逻辑时，禁止以图方便为由进行大面积的代码糅合与侵入式硬编码。
* **按服务解耦标准设计**：每一处新增的业务逻辑（如音视频调度、规则过滤器、变量状态同步等），其函数和模块定义必须遵从"将来可以随时被抽离为独立的微服务插件（Services / Plugins）"的解耦契约标准，实现功能逻辑的高内聚，为长线演进的"微内核沙盒"架构预留平滑抽离的通道。
* **单文件行数硬上限**：除非涉及自动生成的第三方绑定代码（如 `mvu_bundle.js` 等 minified bundle），严禁任何单个 `.ts` 或 `.tsx` 文件超过 **1000 行**。当文件接近此阈值时，必须按职责边界拆分为独立模块，通过 barrel 文件（`index.ts`）重新导出公共 API，确保外部消费者的导入路径不变。
