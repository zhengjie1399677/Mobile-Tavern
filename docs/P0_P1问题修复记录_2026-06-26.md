## 二、P0 级问题修复详情

### P0-1：ChatContext 全量加载 sessions 导致启动卡顿

**根本原因**：`ChatContext` 在应用启动时调用 `getAllSessions()` 全量反序列化整个 sessions 表，当会话记录积累到数百条时，反序列化耗时导致启动白屏。

**修复方案**：
- IndexedDB 升级至 v7，为 `sessions` store 新增 `createdAt` 索引
- 新增 `getSessionsCount()` 与 `getSessionsPaginated(page, pageSize)` 方法，利用 `createdAt` 索引倒序遍历
- `ChatContext` 改为分页加载：首屏加载 50 条，滚动到底部时 `loadMoreSessions` 增量加载
- `hasMoreSessions` 判定改为本页返回数量是否达到 `pageSize`，避免闭包旧值

## 三、P1 级问题修复详情

### P1-1/2/3：服务生命周期 init/destroy 缺失

**根本原因**：9 个内核服务（DatabaseService/LLMService/PromptService/TelemetryService/ScriptService/ChatStreamService/MultiMessageService/TableMemoryService/AutoSummaryService）未实现 `init(kernel, signal?)` 与 `destroy()` 生命周期钩子，导致：
- 服务初始化挂起时无超时熔断（P1-1）
- 服务级 `AbortController` 在内核销毁时不被回收（P1-2）
- `index.ts` 未配置 `initTimeoutMs`（P1-3）

**修复方案**：
- 9 个服务统一补全 `init(kernel, signal?)` 与 `destroy()` 模式：
  ```typescript
  private abortController: AbortController | null = null;
  init(kernel: IKernel, signal?: AbortSignal): void {
    this.kernel = kernel;
    this.abortController = new AbortController();
    if (signal) {
      if (signal.aborted) this.abortController.abort();
      else signal.addEventListener("abort", () => this.abortController?.abort());
    }
  }
  destroy(): void {
    this.abortController?.abort();
    this.abortController = null;
  }
  ```
- `index.ts` 为所有服务配置 `initTimeoutMs`（Database=5000, LLM/AutoSummary=8000, ChatStream=5000, 其余=3000）

**修改文件**：9 个服务文件 + [src/kernel/index.ts](file:///d:/projects/Mobile-Tavern/src/kernel/index.ts)

### P1-7：ChatStreamService 后台流泄漏

**根本原因**：`ChatStreamService.streamLlmResponse` 的 async generator 在消费方提前退出时（如 `for await` break），后台 `readSSEStream` 仍在运行，未被主动取消。

**修复方案**：
- `readSSEStream` 的 `SSEStreamOptions` 新增 `signal?: AbortSignal` 字段
- 在 `readSSEStream` 内部注册 `onSignalAbort` 监听器，signal 触发时立即 `reader.cancel()`
- `finally` 块中 `removeEventListener` 避免内存泄漏
- `ChatStreamService.streamLlmResponse` 创建 `streamAbortController`，传入 `readSSEStream`
- generator 主体包裹 `try/finally`，`finally` 中调用 `streamAbortController.abort()` 主动取消后台流

**修改文件**：
- [src/utils/streamReader.ts](file:///d:/projects/Mobile-Tavern/src/utils/streamReader.ts) — 接收 signal
- [src/kernel/services/ChatStreamService.ts](file:///d:/projects/Mobile-Tavern/src/kernel/services/ChatStreamService.ts) — generator finally abort

### P1-8：Bison 连续推进的 setTimeout 无清理机制

**根本原因**：野牛模式触发后通过 `setTimeout(handleSendMessage, 500)` 递归调用，未保存 timer id，未在组件卸载或会话切换时清理。组件卸载后仍会触发，对已卸载组件 state 进行更新；快速切换多角色可能堆积多个 setTimeout 竞态。

**修复方案**：
- `useChatUI` 新增 `bisonChainTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null)`
- `useSendMessage` 将 `setTimeout` 返回值存入 `bisonChainTimerRef`
- 调度新 timer 前先清理可能残留的旧 timer，避免堆积竞态
- 在四个清理点完整回收：
  1. **组件卸载**：`useChatUI` 的 cleanup effect 中 `clearTimeout`
  2. **会话/角色切换**：`useChat` 的 abort effect 中 `clearTimeout`
  3. **手动停止**：`handleStopGeneration` 中 `clearTimeout`
  4. **调度前防堆积**：新 setTimeout 前检查并清理旧 timer

**修改文件**：
- [src/hooks/useChat/useChatUI.ts](file:///d:/projects/Mobile-Tavern/src/hooks/useChat/useChatUI.ts) — bisonChainTimerRef 声明 + 卸载清理
- [src/hooks/useChat.tsx](file:///d:/projects/Mobile-Tavern/src/hooks/useChat.tsx) — abort effect 清理 + p 对象透传
- [src/hooks/useChat/useSendMessage.ts](file:///d:/projects/Mobile-Tavern/src/hooks/useChat/useSendMessage.ts) — timer 存入 ref + handleStopGeneration 清理

### P1-10：角色卡 extensions 透传原型污染键名

**根本原因**：`cardParser` 导入角色卡时直接透传 `extensions` 字段，未递归移除 `__proto__`/`constructor`/`prototype` 等原型污染键名，存在安全风险。

### P1-11：写队列无背压机制

**根本原因**：`enqueueWrite` 全局写队列无深度上限，高频写入场景下队列可无限增长导致内存膨胀。

**修复方案**：
- **key 合并机制**：`enqueueWrite` 新增可选 `key` 参数，同 key 的多个写操作仅保留最新 `operation`，共享同一 Promise。将每 key 队列深度收敛为 1
- **深度上限安全网**：`MAX_WRITE_QUEUE_DEPTH = 100`，超过时上报遥测告警（仍入队保证数据完整性，不丢弃）
- `saveSession` 传入 key `session:${session.id}`
- `saveCharacter` 传入 key `character:${character.id}`
- `saveStoredSettings` 传入 key `settings:user_settings`
- 删除操作不传 key（避免与 save 冲突）
- 新增单元测试 `testWriteQueueKeyCoalescing` 验证合并逻辑

**修改文件**：
- [src/utils/localDB.ts](file:///d:/projects/Mobile-Tavern/src/utils/localDB.ts) — key 合并 + 深度上限
- [tests/run_all_tests.ts](file:///d:/projects/Mobile-Tavern/tests/run_all_tests.ts) — 新增 testWriteQueueKeyCoalescing

### P1-12：ChatHistoryTab 三次 reduce 重复计算

**根本原因**：`ChatHistoryTab` 在渲染路径中对每个 session 调用三次 `reduce` 计算总字符数（>1000 判断、/1000 展示、原始值展示），每次渲染 O(N×M) 开销。

**修复方案**：
- 改用 `useMemo` 预计算 `enrichedSessions`，仅在 `sessions` 或 `characters` 变化时重算
- 每个 session 仅做一次 `reduce` 计算总字符数，并根据阈值格式化为展示串
- 排序、角色查找、最后消息提取均下沉到 `useMemo`

**修改文件**：

## 四、延后问题说明

### P1-4/5/6：角色卡大字段分轨存储

**延后原因**：完整实施需将 avatar base64 从 `characters` store 分流至独立的 `character_assets` store（DB v8 迁移），并配套 CharactersTab/ChatTab 的懒加载 UI 改造。直接修改 `getAllCharacters` 返回轻量记录（移除 avatar）将破坏列表视图头像显示，属于侵入性架构变更，超出 P1 修复范畴。

**已缓解**：P0-4 的 `getCharacterById` 基础设施已建立，AutoSummaryService 不再调用 `getAllCharacters`。

**后续实施方案**（建议专项处理）：
1. DB v8 升级：创建 `character_assets` store（key = character ID, value = `{ avatar: string }`）
2. `saveCharacter`：avatar 写入 assets store，主记录清空 avatar 字段
3. `getCharacterById`：合并主记录 + assets store 的 avatar（向前兼容：fallback 到主记录）
4. `getAllCharacters`：返回轻量记录（无 avatar）
5. `CharacterContext`：`activeCharacter` 改为通过 `getCharacterById` 异步获取完整数据
6. `CharactersTab`：列表项懒加载 avatar（IntersectionObserver 或按需 `getCharacterAvatar(id)`）
7. 迁移逻辑：DB v8 `onupgradeneeded` 中遍历现有角色卡，将 avatar 复制至 assets store

## 七、风险评估与上线建议

### 已修复问题风险评估

所有修复均通过 `npm run lint` 类型检查与 `npm run test` 全量测试验证，未引入回归问题。关键修复点的风险控制：

- **P0-1 分页加载**：`hasMoreSessions` 判定改为本页返回数量判断，避免闭包旧值导致的无限加载 bug
- **P0-3 防腐层**：`cleanLLMResponse` 对 null/undefined 输入安全降级，对缺少 choices 的响应不崩溃
- **P1-8 Bison timer**：在四个清理点完整回收，调度前防堆积，避免对已卸载组件 state 进行更新
- **P1-11 写队列合并**：删除操作不参与合并（避免与 save 冲突），深度上限不丢弃写入（保证数据完整性）

### 延后问题风险

P1-4/5/6（角色卡大字段分流）延后不影响核心功能，关键性能点已由 P0-4 的 `getCharacterById` 缓解。建议作为独立专项任务实施，配套 UI 懒加载改造与 DB v8 迁移。

### 上线建议

当前状态满足上线标准。建议在发布前进行一次完整的 Android 真机端到端验证，重点覆盖：
1. 会话分页加载与增量加载（P0-1）
2. 野牛模式连续推进与手动停止（P1-8）
3. 角色卡导入与保存（P1-10/P1-13）
4. 流式发送与中断（P1-7）
